application.properties 생성 및 선언하기

main - resources에 application.properties를 생성하고 변수와 변수 값을 선언합니다.


선언한 변수 출력하기(15순위)

그리고 선언한 변수를 출력하기 위해 ApplicationRunner에 @Value 를 사용하여 선언하고 변수를 만들어줍니다. 그리고 override한 run 메소드에 선언하고 출력합니다.

내가 선언한 외부 설정이 출력된 것을 볼 수 있습니다.


커맨드 라인 출력하기(4순위)

packaging을 한 jar를 커맨드 라인을 선언하고 출력하면 우선순위가 높기 때문에 커맨드 라인 값으로 출력됩니다.


외부 설정 테스트하기

테스트를 하기 위해선 test 경로에 있는 class에서 작업을 진행합니다.

Enviroment를 의존성 주입하여 테스트에 사용합니다.
그리고 비교 테스트를 위해선 assertThat 함수가 필요한데 이를 사용하기 위해서는 static으로 import해야 사용할 수 있습니다.
getProperty를 통해 변수를 가져오고 isEqualTo 내장함수로 값이 같은지 비교합니다.
이상 없이 테스트가 완료될 것입니다.

테스트를 위하여 따로 application.properties를 만들기 위해서는 resources 폴더를 생성하여 내부에 새롭게 만들어주어야 합니다.
intelli J 에서는 위와 같이 만들면 됩니다.

테스트 properties에 테스트와 동일한 값을 넣고 비교하면 오류 없이 출력됩니다.


@SpringBootTest로 출력하기(3순위)

이 어노테이션의 장점은 바로 작성하여 테스트할 수 있어 신속하게 테스트가 가능합니다.

properties를 바로 비교한 모습입니다.


@TestPropertySource로 출력하기(2순위)

이 어노테이션의 장점은 테스트시 순위가 가장 높아 확실한 값을 얻을 수 있어 디버깅 하기 용이합니다.

테스트가 출력된 모습입니다.

하지만 한 두개의 값이면 다행이지만 수많은 변수와 값이 있다면 사실상 main과 test의 properties를 언제나 동일하게 유지할 수 없습니다.
그래서 대표적으로 2개의 방법을 사용하여 테스트를 진행합니다.


테스트시 properties 편리하게 관리하기

1. test 폴더에 있는 application.properties를 삭제한다.

이 방법을 이해하기 위해선 어플리케이션이 실행되는 원리를 알아야합니다.

test에 있는 소스들은 전부 main에 있는 소스들에 의해 오버라이딩됩니다. 그렇기 때문에 만약 main에 있는 변수가 test에 없다면 동일하게 선언해주어야 하는 것입니다.

때문에 test에 properties를 없애면 main에 있는 값으로 진행되기 때문에 없어도 되는 것입니다.


2. classpath를 지정하여 따로 관리한다.

위처럼 동일한 이름이 아닌 새로운 이름으로 생성하여 classpath를 지정하여 해당 properties만 사용하는 것입니다.

기타 기능들 활용하기

  • 플레이스 홀더
